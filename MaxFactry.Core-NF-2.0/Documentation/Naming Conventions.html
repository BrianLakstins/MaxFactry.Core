Brad Abrams: Internal Coding Guidelines

.NET Framework Guidelines for Names

They should be followed for cases that are not addressed below.

1. Style

1.1 Tabs & Indenting:  Use Tabs - don't use spaces.  You can then set your editor to the amount of spacing you need to view code (some prefer 2 spaces per tab - others 4 or 5).

1.2 Bracing: Always use braces.  All of the formats below are fine.  Example 1.2.2 is preferred.

Example 1.2.1:
if (someExpression)
{
   DoSomething();
}
else
{
   DoSomethingElse();
}
Example 1.2.2:
if (someExpression){
   DoSomething();
}
else{
   DoSomethingElse();
}
Example 1.2.3:
if (someExpression){   DoSomething();}else{  DoSomethingElse();}

1.3 Comments: Comments should be used to describe intention, algorithmic overview, and/or logical flow.

1.3.1 Copyright notice:  If possible, each file should start with a copyright notice with the last year the file was edited as the copyright year.

1.3.2 Change log: Each file should have a "Change Log" comments after the copyright notice.  Each entry tin the "Change Log" area should include the date of the change, the author, and a comment.

Example 1.3.2.1
/***********************************************************************************
 *
 *  2009.9.18.0 [Brian A. Lakstins] Initial release
 *
 ***********************************************************************************/

1.3.3 Documentation Comments:  All classes and methods should have XML doc comments.  For internal dev comments, the <devdoc> tag should be used.

1.4 Spacing: Visual Studio automatically spaces code if given the chance.  The default spacing seems to be easy to read.  These are some cases:

  • Do use a single space after a comma between function arguments.
    Right:          Console.In.Read(myChar, 0, 1);
    Wrong:       Console.In.Read(myChar,0,1); 
  • Do not use a space after the parenthesis and function arguments
    Right:          CreateFoo(myChar, 0, 1)
    Wrong:       CreateFoo( myChar, 0, 1 )
  • Do not use spaces between a function name and parenthesis.
    Right:          CreateFoo()
    Wrong:       CreateFoo ()
  • Do not use spaces inside brackets.
    Right:    x = dataArray[index];
    Wrong:       x = dataArray[ index ];
  • Do use a single space before flow control statements
    Right:          while (x == y)
    Wrong:       while(x==y)
  • Do use a single space before and after comparison operators
    Right:          if (x == y)
    Wrong:       if (x==y)
1.5 Naming: 

1.5.1 Libraries
  • All identifiers should use Pascal Casing (The first letter in the identifier and the first letter of each subsequent concatenated word are capitalized.)
  • Prefix Interfaces with "I"
  • Class internal variables should be prefixed with two characters.  The first character is "_" and the second character depends on the type of the variable.
  • Variables that only exist in the context of the current method (parameters or local variables) should be prefixed with two characters.  The first character is "l" and the second character depends on the type of the variable.
  • The basic secondary characters that depend on the type of the variable are
    • o -> Object
    • n -> Number (int, double, etc)
    • s -> String
    • a -> Array (List, Dictionary, etc)
    • b -> Boolean
    • e -> Enum
    • d -> DateTime
  • Identifiers should be prefixed in some way to make it obvious they are not part of the core .NET Framework or external library.
1.5.2 WebSites
  • All server controls defined in .ASPX files will have "ID" property that begins with "uc" for "User Control" (ie: ucBodyLabel)
  • All server controls defined in .ASPX files will have "SkinID" property that starts with "us" for "User Skin" (ie: usBodyLabel)
  • CSS Class names will start with "uss" for User Style Sheet  (ie: ussBodyLabel)
  • Domain Object Model objects defined in client side "id" fields will be prefixed with "ud" (ie: udBodyLable)
  • Objects defined in App_Code to be used by the User Interface will begin with "UI" for (ie: UIPage)
  • Objects defined in App_Code to be used in the Business Logic will begin with "BL" for (ie: BLCatalog)
  • Javascript objects created to be used in the browser will be prefixed in some way to make it obvious they are not part of the core DOM or external library.
  • Javascript objects created to be used in the browser will be suffixed by "JS" (ie: BalCatalogJS)
  • Instances of Javascript objects will use the two character prefix naming convention for libraries, with "j" as the first character (ie: joCatalog, jsTitle)
